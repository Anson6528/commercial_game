# 结算后端化正式说明

## 目标

本阶段将结算流程从“前端展示 + 后端记分”提升为“后端正式结算 + 归档留痕 + 幂等完成”。

正式化后的目标包括：

- 结算分数由后端统一裁定
- 同一局结算只完成一次
- 结算结果拥有可追踪的记录号与完成时间
- 排行榜成绩写入与结算归档处于同一后端完成动作中
- 刷新、重复打开结算页时，优先返回已归档结果而不是重新计算临时值

## 当前接口

### 1. 结算预览

`GET /api/session/{session_id}/settlement`

用途：

- 会话尚未正式完成时，返回实时计算的结算结果
- 会话已经完成归档时，直接返回归档记录

响应要点：

- `ok`
- `settlement`
- `archived`

当 `archived = true` 时，`settlement` 内将带有：

- `recordId`
- `finalizedAt`
- `accountId`

### 2. 正式完成结算

`POST /api/session/{session_id}/complete`

请求体：

```json
{
  "accountId": "acc-123"
}
```

用途：

- 统一执行正式结算
- 如提供 `accountId`，同步尝试刷新排行榜成绩
- 生成并保存结算归档记录

响应要点：

- `ok`
- `archived: true`
- `settlement`
- `account`

该接口设计为幂等：

- 同一 `session_id` 已完成过结算时，不重复生成新记录
- 后续重复请求将返回原有归档结果

### 3. 清理进行中会话

`DELETE /api/session/{session_id}`

用途：

- 玩家回大厅
- 玩家重新开局
- 玩家登出

说明：

- 该操作只清理进行中的会话缓存
- 已归档的结算记录不会被删除

## 归档数据

当前新增归档表：

- `settlement_records`

主要字段：

- `record_id`
- `session_id`
- `account_id`
- `final_score`
- `result_code`
- `settlement_json`
- `created_at`

约束：

- `session_id` 唯一，保证同一局只归档一次

## 前端行为调整

结算面板打开后：

1. 前端调用 `completeSettlement`
2. 后端完成正式结算并返回归档结果
3. 前端展示后端返回的正式结算数据
4. 如有账号，前端同步拿到更新后的账号最佳成绩

刷新后：

- 若该局已归档，结算查询优先返回归档结果
- 不再依赖前端再次本地拼装分数作为最终依据

## 验证范围

本次已覆盖：

- 结算预览接口
- 已归档结果回读
- 正式完成结算接口
- 会话删除接口
- 前端 TypeScript 类型检查

## 当前边界

本阶段已实现“正式结算归档”与“排行榜写入并轨”，但暂未扩展：

- 多维排行榜分类
- 结算记录后台管理查询页
- 账号安全强化
- 归档记录关联更多行为明细

## 结论

至此，结算链路已从“可用”推进为“更正式”的后端主导模式：

- 有统一裁定
- 有正式归档
- 有幂等完成
- 有刷新回读
- 有排行榜同步
